3. Compound assignment and increment/decrement
+=, -=, *=, /=, ++, --  (add a dedicated node to the AST to preserve the distinction for later codegen).
4. A real semantic-analysis pass
This is the big structural one you've been deferring correctly. Right now nothing checks: undeclared identifiers, return outside a function (the TODO comment sitting in your code), break/continue outside a loop, type mismatches, calling something that isn't a function, wrong argument counts. You'd walk the AST post-parse with a symbol table (scope-per-block, parent pointers for lookup) and a type-checking pass. This is where "is this program actually valid" lives, separate from "is this program grammatically well-formed" which is all the parser guarantees today. I'd build this before going further on parser features, since several things below depend on it.
5. Array/struct literals and initializer syntax
arr : int[3] = { 1, 2, 3 }; and p : Point = { 1, 2 }; — you have array types now but no way to initialize one compactly. This needs a new primary-expression form (probably NODE_ARRAY_LITERAL/NODE_STRUCT_LITERAL, parsed off a leading { in expression position) — mildly tricky because { is already meaningful as a block-start, so you'd lean on context (you're inside an initializer position, not a statement position) to disambiguate, no backtracking needed.
6. const / compile-time value bindings via bare ::
Right now :: only ever leads into ( for procs. Jai/Odin also use :: for compile-time constants: MAX :: 100;. Small addition to parse_decl_after_name — if what follows isn't (, try parsing it as a constant-value expression instead of erroring immediately.
7. A minimal codegen backend
Once semantic analysis exists, the natural payoff is generating something executable — even a simple tree-walking interpreter first (fastest to get "does my language actually run programs" feedback), then maybe LLVM IR or a small bytecode VM later. I'd genuinely do the interpreter before anything else here — it's the cheapest way to validate that everything above (types, scoping, control flow) is actually coherent, since bugs in semantic analysis often only surface once you try to execute the thing.
8. Test infrastructure
Given how much back-and-forth we've had catching regressions by eyeballing error output, this is worth doing soon rather than late: a small harness that feeds .mylang snippets through the parser and diffs against expected ast_pretty_print output (or expected diagnostic messages) would catch the parse_program-bypassing-parse_statement class of bug instantly instead of via a 43-line error dump.
